# 测试规则（防假绿）⭐

### 规则 5：禁止只 mock 不测真链路

**描述**：核心流程必须跑真实环境，不能只用 mock 数据验证。

**验证方式**：人工确认跑过真实接口。

**核心流程**：
- 加载收藏夹（真实 Bilibili API）
- 播放音频（真实音频 URL）
- 切歌（真实切换 + 播放）
- 收藏操作（真实 store 写入）

**允许 mock 的场景**：
- 单元测试中测试纯函数逻辑
- 测试错误处理路径（模拟网络错误）

### 规则 6：修改后必须跑相关测试

**描述**：修改后必须跑一遍相关测试，不能只看单测。

**验证方式**：CI 通过或手动跑过。

**检查清单**：
- [ ] `npx tsc --noEmit` 通过
- [ ] `npm run build` 成功（重大变更时）
- [ ] 手动测试修改涉及的功能

### 规则 7：先确认业务逻辑再看测试结果

**描述**：测试"变绿"之前，先确认业务逻辑没问题。

**验证方式**：人工看一眼。

**常见假绿场景**：
- 测试通过但实际功能不工作
- 测试覆盖了代码但没覆盖业务逻辑
- 测试用例本身有 bug
- 依赖的外部服务返回了非预期数据

**检查方法**：
1. 先读代码，理解业务逻辑
2. 再看测试，确认测试覆盖了关键路径
3. 最后看测试结果
