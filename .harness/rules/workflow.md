# 工作流规则（防提前完成）

### 规则 8：先拆解再编码

**描述**：先做需求拆解（3~5 条子任务），再做编码。

**验证方式**：记录在 `changes/` 里。

**流程**：
1. 在 `changes/<date>-<topic>/` 创建目录
2. 写一个简单的任务清单（3~5 条）
3. 每条任务标注：做什么、改哪个文件、怎么验证
4. 开始编码

**模板**：见 `templates/task-checklist.md`

### 规则 9：验证通过再报告完成

**描述**：自己验证通过后再报告"完成"。

**验证方式**：附上验证截图/日志。

**完成报告必须包含**：
- 做了什么（1-2 句话）
- 改了哪些文件
- 验证结果（tsc 输出、build 日志、或手动测试步骤+结果）

**反例**：
```
"已完成"  ← 没有验证结果，不算完成
```

**正例**：
```
"已完成。修改了 FloatingPlayer.tsx 的拖拽逻辑。
验证：npx tsc --noEmit 通过，npm run build 成功，
手动测试拖拽移动窗口正常。"
```

### 规则 10：复杂任务拆多次会话

**描述**：复杂任务拆成多次会话，不要挤在一个长会话里。

**验证方式**：每次只做一件事。

**判断标准**：
- 涉及 3 个以上文件 → 拆分
- 需要多次手动验证 → 拆分
- 会话已超过 30 分钟 → 考虑拆分

**拆分方法**：
```
会话 1：需求分析 + 任务拆解
会话 2：核心功能实现
会话 3：测试 + 修复
会话 4：文档更新 + 归档
```
